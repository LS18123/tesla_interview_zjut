特斯拉跨年财报智能问答系统 (Tesla-Financial-RAG)

本项目针对 2021-2025 年特斯拉 10-K 与 10-Q 财报构建，旨在解决复杂财务审计场景中的跨年对比、多步推理及数字计算难题 。系统采用了 父子索引（Parent-Document Retrieval, PDR）架构，有效提升了财务报表场景下的检索精度与上下文完整性 。

第一部分：项目背景与设计目标

1.1 背景分析

信息密度极高：单份财报动辄上百页，包含大量非结构化文本与复杂财务表格 。

结构规范但复杂：包含 MD&A、财务报表附注、风险因素等特定章节 。

时间关联性强：财务分析需对比多个季度、多个年份的趋势 。

1.2 核心挑战

跨文档检索：要求系统具备从多个年份文档中同时提取并关联信息的能力 。

表格数据失真：防止 PDF 按文本切分导致行列关系断裂，确保准确的数字读取 。

高精度术语检索：解决财务领域高度相似术语（如 Gross Margin）的检索偏差问题 。

多步推理需求：处理涉及数字寻找、对比排序及文本解释的多跳场景 。

第二部分：系统架构与核心工程实现

2.1 环境管理

工具选择：全面采用 uv 作为环境管理工具 。

性能优势：利用 uv sync 实现毫秒级的依赖解析与环境同步 。

版本确定性：通过 uv.lock 锁定所有底层依赖版本，确保环境高度一致 。

2.2 深度文档解析策略 (Parser)

上下文注入：在每页头部硬编码注入包含年份、页码、章节的“上下文信封” 。

会计周期判定：利用正则表达式强制锁定封面日期，确保 Year 和 Quarter 元数据准确 。

物理布局保护：通过 page.outside_bbox() 逻辑预先排除表格区域，实现图文分离 。

表格序列化：将表格统一转换为 Markdown 格式，保留关键行列索引关系 。

2.3 父子索引架构 (PDR Chunking)

子块 (Child Chunks)：将页面内容切分为 400 字符的小块用于向量化，精准捕捉指标关键词 。

父文档 (Parent Store)：以“整页”作为父文档单元，命中子块后自动回溯提取完整页面上下文 。

设计优势：从根本上解决了财务表格被“拦腰斩断”导致的计算错误 。

2.4 检索引擎与推理逻辑

语义引擎：选用支持 8192 Tokens 输入的 BAAI/bge-m3 模型，提升语义表征效果 。

元数据过滤：利用 filter={"year": str(year)} 实现先验过滤，杜绝跨年数据误读 。

意图规划 (Intent Planner)：利用 LLM 对问题进行语义拆解，输出结构化 JSON 审计计划 。

强制溯源原则：要求模型在生成答案时必须标注具体页码来源（如 [Source: Year, Page]） 。

第四部分：失败案例剖析

本项目对测评中识别出的典型失败案例进行了深度溯源（详细分析见 FAILURE_ANALYSIS.md）：

案例 1：风险因素 (Item 1A) 提取覆盖不足。由于检索策略过度依赖硬过滤，导致泛语义词汇唤起失效。改进方案为引入章节感知检索 。

案例 2：多维度数值排序逻辑偏差。由于模型在长文本中存在“首因效应”，导致数值比较错误。改进方案为增加中间思考层强制执行排序逻辑 。

第五部分：总结与展望

5.1 项目总结

系统通过 PDR 架构 成功解决了财务 RAG 中核心的语义断裂痛点，实现了极高的审计透明度，所有答案均可溯源至原始财报页面 。

5.2 未来优化方向

多模态解析增强：引入 Vision-based OCR 针对极复杂嵌套表格进行像素级解析 。


代码解释器集成：设计模型自动生成 Python 代码执行数值计算，消除运算不稳定性 。


长上下文架构升级：迁移至具备更高 Token 容量的模型，以容纳更多年份的父文档上下文 。
